# 经典 K 线识别引擎设计说明

本文档说明 Jarvis-Quant 中“经典 K 线识别推荐”能力的设计逻辑、训练流程、准确性保障机制，以及后续自动演进方案。

当前实现对应核心文件：

- `backend/etl/utils/kline_patterns.py`
- `backend/etl/utils/patterns.py`
- `backend/api/admin.py`
- `scripts/train_kline_patterns.py`

## 1. 目标与定位

该模块不是一个“见形态就买”的静态图谱库，而是一个面向 A 股日线场景的结构化识别引擎。

它的职责分为 4 层：

1. 识别日线级经典 K 线形态。
2. 结合趋势、位置、量能、波动上下文，对形态进行置信度打分。
3. 使用历史日线数据回测各形态的真实效果，对实时信号做历史校准。
4. 为后续插件推荐、持仓点评、卖点建议提供标准化信号输入。

该引擎输出不是最终买卖建议，而是“经典形态证据层”。后续必须与趋势因子、资金因子、主线强度、市场环境一起融合，才能形成最终推荐。

## 2. 设计原则

### 2.1 避免纯阈值误判

传统 K 线识别常见问题是：

- 只看单日实体和影线比例。
- 不看所处趋势。
- 不看高低位位置。
- 不看量能确认。
- 不做历史样本验证。

因此本项目将识别拆成两段：

1. `结构判定`
2. `上下文确认`

只有二者同时满足，才输出高置信度信号。

### 2.2 严格避免未来函数

实时识别时，所有规则只能使用“当前及之前”的数据。

允许使用未来数据的地方只有两个：

- 历史回测评估
- 训练校准表

这两部分只服务于参数校准和信号再加权，不参与当日形态的原始识别条件。

### 2.3 兼容现有链路

当前系统已经有两条链路依赖形态识别：

- Watchlist 专业点评链路
- `PatternRecognizer` 的兼容接口

因此本次升级保持对外接口不变：

- `detect_all_patterns(df)`
- `get_latest_signals(df, min_confidence=...)`
- `PatternRecognizer.recognize()`

这样后续接入 `base.py` 或新的 Falcon 插件时，不需要重写前端和已有点评逻辑。

## 3. 整体架构

整体流程如下：

```text
daily_price 日线历史
    ->
基础特征工程
    ->
经典形态识别
    ->
原始置信度
    ->
历史样本回测评估
    ->
校准文件 kline_pattern_calibration.json
    ->
实时识别时读取校准文件
    ->
输出校准后置信度 + 历史命中信息
    ->
专业点评 / 插件推荐 / 卖点建议
```

## 4. 核心特征工程

引擎在 `backend/etl/utils/kline_patterns.py` 中统一构建以下特征。

### 4.1 K 线结构特征

- 实体上沿 `body_top`
- 实体下沿 `body_bottom`
- 实体大小 `body_size`
- 上影线 `upper_shadow`
- 下影线 `lower_shadow`
- 总振幅 `total_range`
- 实体占比 `body_ratio`
- 收盘位置 `close_pos`
- 开盘位置 `open_pos`

### 4.2 波动归一化特征

- `true_range`
- `atr14`
- `body_to_atr`
- `range_to_atr`
- `body_vs_avg`
- `range_vs_avg`

作用：

- 解决“同样 3% 实体，在低波动股和高波动股上含义完全不同”的问题。
- 让形态识别不再依赖固定价格尺度。

### 4.3 趋势与位置特征

- `ma5 / ma10 / ma20 / ma60`
- `ma5_slope_3 / ma10_slope_5 / ma20_slope_5`
- `ret_3 / ret_5 / ret_10 / ret_20`
- `high_20 / low_20`
- `close_rank_20`
- `near_low_20 / near_high_20`
- `trend_up / trend_down`

作用：

- 区分“底部锤子线”和“高位震荡线”。
- 区分“启动型红三兵”和“高位衰竭红三兵”。

### 4.4 量能特征

- `vol_ma5 / vol_ma10 / vol_ma20`
- `vol_ratio_5 / vol_ratio_10 / vol_ratio_20`
- `vol_trend`

作用：

- 验证突破型形态是否有资金确认。
- 弱化无量形态误报。

## 5. 形态识别逻辑

目前已纳入主引擎的经典形态包括：

### 5.1 看涨形态

- 锤子线 `BULLISH_HAMMER`
- 看涨吞没 `BULLISH_ENGULFING`
- 曙光初现 `PIERCING_LINE`
- 启明星 `MORNING_STAR`
- 红三兵 `THREE_WHITE_SOLDIERS`
- 上升三法 `RISING_THREE_METHODS`
- 仙人指路 `IMMORTAL_GUIDE`
- 老鸭头 `OLD_DUCK_HEAD`
- 量价齐升 `ACCUMULATION`

### 5.2 看跌形态

- 上吊线 `HANGING_MAN`
- 射击之星 `SHOOTING_STAR`
- 看跌吞没 `BEARISH_ENGULFING`
- 乌云盖顶 `DARK_CLOUD_COVER`
- 黄昏星 `EVENING_STAR`
- 三只乌鸦 `THREE_BLACK_CROWS`
- 下降三法 `FALLING_THREE_METHODS`
- 量价背离 `DIVERGENCE`

### 5.3 中性/警示形态

- 十字星 `DOJI`
- 地量地价 `EXTREMELY_LOW_VOLUME`

### 5.4 识别框架

每个形态都遵循同一个框架：

1. `结构条件`
2. `趋势条件`
3. `位置条件`
4. `量能确认`
5. `置信度合成`

例如锤子线不再只是：

- 小实体
- 长下影

而是必须进一步确认：

- 近期处于下跌或低位区域
- 当日振幅有意义，不是噪音小波动
- 收盘不能太弱
- 量能放大时置信度更高

这样做的目的是减少“所有长下影都被识别成锤子线”的误判。

## 6. 置信度模型

### 6.1 原始置信度

每个形态先输出 `0.0 ~ 1.0` 的原始置信度。

规则：

- 不满足结构条件时为 `0.0`
- 满足基本结构时进入 `0.5 ~ 1.0`
- 结构越标准、上下文越匹配、量能越确认，分值越高

当前采用的统一合成函数是：

```text
final_conf = 0.5 + 0.5 * mean(component_scores)
```

其中 `component_scores` 包括结构质量、趋势匹配、位置匹配、量能确认等分量。

### 6.2 历史校准后的置信度

原始置信度反映的是“像不像这个形态”，但不等于“未来胜率有多高”。

因此实时输出时可以进一步结合历史训练结果：

```text
adjusted_conf = 0.75 * raw_conf + 0.25 * historical_hit_rate
```

实时信号可附带：

- `raw_confidence`
- `confidence`
- `historical_hit_rate`
- `historical_avg_ret`

这样前端和策略层可以同时看到“形态标准度”和“历史有效性”。

## 7. 历史训练与回测

### 7.1 训练目标

训练不是为了用机器学习拟合蜡烛图像，而是为了回答两个问题：

1. 哪些形态在 A 股日线历史上真正有效。
2. 同一形态在不同置信度区间下，未来表现是否显著不同。

### 7.2 训练输入

训练数据来自 `daily_price`，最少需要字段：

- `trade_date`
- `ts_code`
- `open`
- `high`
- `low`
- `close`
- `vol`
- `amount`

### 7.3 训练流程

训练脚本：

```bash
python3 scripts/train_kline_patterns.py \
  --start-date 2023-01-01 \
  --end-date 2026-03-07 \
  --horizons 3,5,10 \
  --min-confidence 0.55
```

训练步骤：

1. 读取 `daily_price` 历史。
2. 按 `ts_code + trade_date` 排序。
3. 对每只股票独立运行 `detect_all_patterns()`。
4. 为每个触发信号计算未来 `3/5/10` 日收益。
5. 统计各形态样本数、命中率、平均收益、方向收益。
6. 按置信度桶分组，例如 `50-65 / 65-80 / 80-100`。
7. 生成校准文件 `kline_pattern_calibration.json`。

### 7.4 评估指标

训练时重点关注：

- `sample_count`
- `avg_confidence`
- `hit_rate_3d / 5d / 10d`
- `avg_ret_3d / 5d / 10d`
- `directional_edge_3d / 5d / 10d`
- `ret_std`

其中 `directional_edge` 的定义更适合策略视角：

- 看涨形态：直接使用未来收益
- 看跌形态：使用未来收益的相反数

这样不同方向的形态可以放在同一排序体系里比较。

### 7.5 贝叶斯平滑

训练中加入了先验平滑，防止小样本形态被误判为“高胜率”。

原因：

- 某形态可能只出现 8 次，但其中 7 次成功。
- 直接用 `87.5%` 作为命中率会严重高估。

因此系统使用：

- `raw hit rate`
- `bayes hit rate`

后者会把方向先验纳入计算，对样本量小的形态自动降权。

## 8. 如何保证准确性

### 8.1 结构准确性

通过以下方式保证“识别的是经典形态本身”：

- 不只使用单一比例阈值
- 使用 ATR 与历史均值做归一化
- 使用趋势、位置、量能做二次确认
- 多 K 线组合形态要求更完整的相对关系

### 8.2 样本准确性

通过以下方式保证“不是纸上谈兵”：

- 所有形态都可以回放到历史真实日线
- 每个形态都能拿到未来收益验证
- 每次训练都输出样本数和命中率

### 8.3 推荐准确性

系统明确区分：

- `形态识别准确`
- `交易建议准确`

前者只负责识别图形与历史有效性。

后者还必须叠加：

- 大盘环境
- 主线强弱
- 个股流动性
- 资金行为
- 换手与波动风险

因此后续插件层必须把 K 线形态当作因子之一，而不是最终决策器。

### 8.4 避免过拟合

为避免“只对某一段行情有效”，建议训练时使用：

- 多年份样本
- 牛市、震荡市、下跌市混合样本
- 至少滚动 2~3 年窗口

并对以下现象保持警惕：

- 某形态高胜率但样本过少
- 某形态只在极端市场阶段有效
- 某形态 3 日有效但 10 日无效

## 9. 后续自动演进方案

### 9.1 演进目标

后续系统不应靠手工不断改阈值，而应形成自动迭代闭环。

目标闭环：

```text
每日新数据
  ->
周期性训练 / 评估
  ->
生成新校准表
  ->
对比旧版表现
  ->
达到门槛才提升为正式版本
  ->
写入插件与点评链路
```

### 9.2 推荐演进机制

建议引入以下流程：

1. 每周或每月训练一次新的校准文件。
2. 与当前线上版本进行对比评估。
3. 只有在关键指标同时改善时，才允许替换线上校准文件。

推荐门槛：

- 总样本数不下降太多
- 5 日方向命中率提升
- 5 日方向收益提升
- 高置信度区间的稳定性提升
- 小样本形态没有异常飙升

### 9.3 版本化管理

建议把校准文件纳入版本管理：

- `kline_pattern_calibration.v1.json`
- `kline_pattern_calibration.v2.json`

同时记录：

- 训练时间
- 训练区间
- 数据样本数
- 评估指标摘要

### 9.4 灰度发布

建议不要直接全量切换到新校准版本。

更稳妥的做法：

1. Watchlist 点评先使用新版本观察 1 周。
2. 插件推荐同时输出旧版和新版评分对比。
3. 当新版在历史复盘和实际观察中都更稳定，再正式切换。

### 9.5 自动淘汰机制

如果某些形态长期表现恶化，应自动降权甚至停用。

例如满足以下任一条件即可进入观察名单：

- 最近 6 个月样本数持续很低
- 5 日方向命中率显著低于方向基准
- 高置信度区间失去区分度
- 与量价和趋势因子出现系统性冲突

## 10. 与后续插件的集成建议

未来“经典 K 线识别推荐”插件接入时，建议不要直接输出“买/卖”，而是输出标准化因子。

推荐插件输出结构：

- `pattern_code`
- `pattern_name`
- `raw_confidence`
- `calibrated_confidence`
- `historical_hit_rate_5d`
- `historical_avg_ret_5d`
- `direction`
- `context_tags`

然后在插件层与其他因子融合，例如：

- 趋势因子
- 放量因子
- 主线板块因子
- 市场环境因子
- 风险控制因子

推荐总分示意：

```text
final_score =
  0.30 * kline_score +
  0.25 * trend_score +
  0.20 * moneyflow_score +
  0.15 * market_regime_score +
  0.10 * risk_penalty
```

## 11. 与 Watchlist 专业点评的集成建议

Watchlist 不适合直接展示底层信号码，而应展示“解释性更强”的专业点评。

建议前端展示优先级如下：

1. 优先展示校准后高置信度形态。
2. 同时展示历史有效性摘要。
3. 对看跌形态加强卖点提示。

示例：

- `启明星，历史 5 日胜率 61%，属于底部反转型信号`
- `射击之星，历史 5 日转弱概率较高，若次日失守 5 日线建议减仓`
- `乌云盖顶 + 量价背离，同步出现时优先作为卖点预警`

## 12. 当前限制

当前版本仍有边界：

- 只覆盖日线，不含分时与周线共振
- 仍然属于规则引擎，不是图像模型
- 对极端题材股、连续一字板个股的解释能力有限
- 对复权方式、停牌缺口、异常波动样本仍需进一步细化

## 13. 下一阶段建议

建议按以下顺序推进：

1. 先用全市场历史日线完成第一次训练与校准。
2. 输出每个形态的 3/5/10 日效果榜单。
3. 选出真正有统计优势的形态作为插件首批输入。
4. 将看跌形态优先接入 Watchlist 卖点建议。
5. 再把高质量看涨形态接入 Falcon 插件做选股因子。

## 14. 操作清单

### 14.1 首次训练

```bash
python3 scripts/train_kline_patterns.py \
  --start-date 2023-01-01 \
  --end-date 2026-03-07 \
  --horizons 3,5,10 \
  --min-confidence 0.55
```

### 14.2 训练完成后的检查项

- 是否生成 `backend/etl/utils/kline_pattern_calibration.json`
- 各形态样本数是否足够
- 高置信度分桶是否明显优于低置信度分桶
- 看涨 / 看跌形态是否与常识一致
- 是否存在单一小样本形态异常冲榜

### 14.3 上线前检查项

- Watchlist 点评是否能正常展示新增形态
- `PatternRecognizer` 是否能自动加载校准文件
- 未训练时是否能自动回退到原始规则输出
- 新版校准是否优于旧版校准
