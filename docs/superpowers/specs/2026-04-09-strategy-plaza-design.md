# 策略广场设计文档

> 日期: 2026-04-09
> 状态: 已确认，待实施

## 1. 背景与目标

当前项目已经具备：

- `盯盘` 页面的高频观察与个股分析链路
- `仪表盘` 页面的情绪、主线与组合建议能力
- 收盘后自动 ETL、主线刷新、情绪重算的调度框架

但缺少一个面向“策略结果归档与验证”的独立工作面。用户希望新增一个与 `盯盘` 并列的 `策略广场` 一级入口，用来承载以下能力：

1. 展示策略在指定日期“新进入观察”的标的列表
2. 支持固定策略池切换，策略由后端代码实现，不做前端编辑器
3. 支持按历史日期查看当日结果
4. 每日 18:00 自动生成当日符合条件的标的列表并落库
5. 基于策略定义的“正式进入观察日”做后续 3 / 5 / 10 个交易日回测
6. 对策略近期回测表现给出简短总结

本次设计强调：

- 简约页面，不做流程看板和解释型大段文案
- 策略可插拔，后续新增策略时尽量不改前端结构
- 所有用户共享同一批策略与同一批结果，不做用户隔离
- 平台不强行规定统一策略模板，允许策略自行定义多阶段时序规则

## 2. 已确认约束

本轮确认后的产品边界如下：

- `策略广场` 必须与 `盯盘` 并列，作为一级 tab
- 首版允许没有任何策略，页面可空态上线
- 策略由后端代码生成与维护，不提供前端创建/编辑策略能力
- 所有用户查看同一套策略和同一批结果
- 首屏默认展示“今日新进入观察的标的”
- 前端不展示策略中间阶段池，只展示最终进入观察的结果
- 策略并非固定回测模板，平台必须支持多阶段、延迟入观察、阶段过滤、提前失效等自定义逻辑

## 3. 非目标

首版明确不做以下能力：

- 用户级自定义策略、参数保存、策略克隆
- 可视化工作流编排器、DSL、低代码策略编辑器
- 中间阶段池的前端展示
- 盘中实时策略扫描
- 与现有 `/backtest_grid`、`/backtest_result`、`/backtest_walkforward` 占位接口复用语义
- 页面上的长篇策略教学、过滤器说明、策略百科

## 4. 现状与接入点

### 4.1 前端现状

当前前端主导航在 `AppLayout` 中仅有：

- `盯盘`
- `仪表盘`
- `设置`

因此 `策略广场` 需要作为新的一级 route 接入，桌面顶栏和移动底栏都要新增入口，不能放在设置页或文档抽屉中。

### 4.2 后端现状

当前后端已有：

- `APScheduler` 定时调度器
- 收盘后统一数据刷新 `perform_daily_data_update()`
- `etl_tasks` 任务队列与后台消费者
- 情绪、主线、组合建议等分析模块

这意味着策略广场不应自建独立调度系统，而应复用：

1. 定时器做每日主调度
2. `etl_tasks` 做手动重跑与补算
3. 现有行情、资金流、因子宽表作为策略计算输入

## 5. 设计总览

策略广场采用“阶段型策略插件 + 统一归档层 + 轻前端展示”的结构。

整体链路：

```text
策略注册表
  -> 每日/手动执行器
  -> 策略代码按指定交易日计算“新进入观察的标的”
  -> 统一写入观察归档表
  -> 后台补全 3/5/10 日回测结果
  -> 汇总近期表现
  -> 前端读取策略列表 / 当日观察结果 / 近期摘要
```

平台只关心三件事：

1. 哪个策略启用、怎么展示
2. 某交易日有哪些标的新进入观察
3. 这些观察结果后续 3 / 5 / 10 日表现如何

平台不试图理解所有策略的内部阶段逻辑。阶段逻辑完全属于策略代码自身。

## 6. 后端架构

### 6.1 新目录建议

新增独立目录：

```text
backend/strategy/plaza/
  __init__.py
  base.py
  registry.py
  service.py
  summarizer.py
  builtin/
    __init__.py
    <strategy_key>.py
```

职责划分：

- `base.py`
  定义策略元信息、观察结果、回测配置等基础模型
- `registry.py`
  负责注册与发现启用策略
- `service.py`
  负责按日期执行策略、落库、补算回测、生成汇总
- `summarizer.py`
  负责把统计指标转成简短总结语句
- `builtin/*.py`
  每个固定策略一个文件，内部可自由实现多阶段规则

### 6.2 策略插件契约

首版策略不做模板化 DSL，而是约定一个足够小的 Python 接口。

每个策略至少需要提供：

- `meta()`
  返回策略 key、名称、排序、是否启用、极简描述
- `run_for_date(trade_date, context)`
  返回该日期“新进入观察”的标的列表

每条观察结果至少包含：

- `strategy_key`
- `trade_date`
- `ts_code`
- `name`
- `observation_date`
- `entry_anchor_date`
- `reason`
- `tags`
- `trace_json`

说明：

- `trade_date` 表示本次运行的目标交易日
- `observation_date` 表示该标的正式进入观察池的日期，通常等于当前运行日期
- `entry_anchor_date` 表示回测起点所属日期，由策略自行决定
- `reason` 为前端主表显示的极简原因
- `trace_json` 供排查和内部调试使用，不要求前端展示

### 6.3 阶段型策略的边界

平台必须允许策略实现如下逻辑：

- 某日触发事件
- 次日或后续若干日过滤
- 第 N 个交易日才正式进入观察
- 观察前即已失效则不入观察
- 回测起点不一定等于触发日，也不一定等于次日

例如：

```text
触发日涨幅达到条件
  -> 次日不能连板
  -> 第 8 个交易日才纳入观察
  -> 从第 8 个交易日定义的锚点开始统计 3/5/10 日表现
```

这类逻辑不得被平台压扁成统一模板。平台只接收“最终进入观察的结果”。

### 6.4 执行服务

`StrategyPlazaService` 负责以下动作：

1. 拉取启用策略列表
2. 对指定交易日逐个执行策略
3. 幂等写入观察结果
4. 记录运行日志与任务结果
5. 扫描历史未完成回测结果，补全 3 / 5 / 10 日表现
6. 生成策略近期摘要

幂等要求：

- 同一 `strategy_key + observation_date + ts_code` 重复执行时应更新而不是重复插入
- 手动重跑历史日期时，允许覆盖旧结果

## 7. 数据模型设计

首版新增四张策略广场专用表。

### 7.1 `strategy_definitions`

作用：保存策略注册信息与展示状态。

核心字段：

- `strategy_key` 主键
- `name`
- `description`
- `enabled`
- `display_order`
- `engine_version`
- `updated_at`

说明：

- 首版策略虽由代码注册，但仍建议镜像入库，便于前端统一读取和后续启停
- `engine_version` 用于标记策略逻辑升级后的结果版本

### 7.2 `strategy_observations`

作用：保存某策略某日“新进入观察”的标的，是策略广场主表的直接数据源。

核心字段：

- `strategy_key`
- `trade_date`
- `observation_date`
- `ts_code`
- `name`
- `reason`
- `tags_json`
- `entry_anchor_date`
- `trace_json`
- `created_at`
- `updated_at`

建议唯一键：

- `strategy_key + observation_date + ts_code`

说明：

- `trade_date` 与 `observation_date` 首版通常一致，但分开保留，避免未来策略产生跨日归档歧义
- `reason` 必须足够短，适合首屏列表直接展示

### 7.3 `strategy_backtest_runs`

作用：保存每条观察结果的 3 / 5 / 10 日回测状态与表现。

核心字段：

- `strategy_key`
- `observation_date`
- `ts_code`
- `entry_anchor_date`
- `entry_price`
- `entry_price_source`
- `status`
- `ret_3d`
- `ret_5d`
- `ret_10d`
- `max_gain_3d`
- `max_gain_5d`
- `max_gain_10d`
- `max_drawdown_3d`
- `max_drawdown_5d`
- `max_drawdown_10d`
- `last_completed_horizon`
- `last_eval_date`
- `error`
- `updated_at`

说明：

- `entry_price_source` 用于记录该策略采用何种锚点价格，例如 `close@entry_anchor_date`、`open@next_trade_date`
- 平台不强制统一入场规则，但必须把实际使用的价格来源写清楚
- `status` 可取 `PENDING` / `PARTIAL` / `COMPLETED` / `FAILED`

### 7.4 `strategy_daily_summaries`

作用：保存策略摘要，供页面底部直接读取。

核心字段：

- `strategy_key`
- `trade_date`
- `window_trade_days`
- `observation_count`
- `completed_count_3d`
- `completed_count_5d`
- `completed_count_10d`
- `win_rate_3d`
- `win_rate_5d`
- `win_rate_10d`
- `avg_ret_3d`
- `avg_ret_5d`
- `avg_ret_10d`
- `median_ret_3d`
- `median_ret_5d`
- `median_ret_10d`
- `avg_max_gain_3d`
- `avg_max_gain_5d`
- `avg_max_gain_10d`
- `avg_max_drawdown_3d`
- `avg_max_drawdown_5d`
- `avg_max_drawdown_10d`
- `summary_text`
- `updated_at`

说明：

- 首版默认使用滚动 `120` 个交易日窗口生成摘要
- 如果样本不足，则使用全部可用样本
- `summary_text` 首版采用规则化总结，不默认依赖 AI

## 8. 回测口径

### 8.1 平台口径

平台不规定“信号日次日开盘买入”的固定模板，而是要求策略显式给出：

- 何时正式进入观察
- 从哪个日期和价格锚点开始统计

因此平台的统一责任是：

1. 根据策略提供的 `entry_anchor_date` 和价格来源解析 `entry_price`
2. 基于交易日历寻找后续第 3 / 5 / 10 个交易日
3. 计算各 horizon 的收盘收益、区间最大上涨、区间最大回撤

### 8.2 首版价格来源支持

首版平台层支持少量标准价格源：

- `close_on_anchor`
- `open_next_trade_day`
- `close_next_trade_day`

策略可在返回结果时声明所用价格源。若后续某策略需要更特殊的入场价，可由平台扩展。

### 8.3 补算机制

对于当天新进入观察的标的：

- 当日可能无法完成 3 / 5 / 10 日回测
- 系统在每日策略任务中顺带扫一遍未完成记录
- 一旦未来交易日数据齐备，就补齐对应 horizon

因此每日策略任务要同时做两件事：

1. 生成今日新进入观察的结果
2. 补全历史待完成回测

## 9. 调度与任务设计

### 9.1 每日自动任务

新增策略广场每日主任务：

- 目标时点：`18:00`
- 任务内容：
  - 选择最近完整交易日
  - 执行全部启用策略
  - 写入观察结果
  - 补算未完成回测
  - 更新摘要

### 9.2 兜底与手动重跑

考虑到收盘后数据同步存在延迟，策略广场需要支持：

- 手动指定历史日期重跑
- 手动指定单个策略重跑
- 当 18:00 数据不完整时，可在后台再次触发补跑

首版建议：

- 定时主任务放进 `APScheduler`
- 手动重跑统一走现有 `etl_tasks` 队列，避免阻塞 HTTP 请求

### 9.3 数据完整性门槛

策略任务在执行前应检查：

- `daily_price`
- `stock_daily_basic`
- `stock_moneyflow`
- `stock_factor_daily`

对应交易日是否具备最低可用数据。

若关键输入缺失：

- 记录失败日志与任务错误
- 不覆盖旧摘要
- 保留页面上一次成功快照

## 10. API 设计

策略广场使用独立命名空间，不复用现有占位回测接口。

### 10.1 `GET /admin/strategy-plaza/strategies`

返回：

- 所有已注册策略
- 是否启用
- 显示顺序
- 最近一次运行日期
- 最近一次是否有结果

说明：

- 最近运行信息首版通过 `strategy_observations` / `strategy_daily_summaries` 推导
- 手动触发任务的执行状态仍复用现有 `etl_tasks`

### 10.2 `GET /admin/strategy-plaza/observations`

参数：

- `strategy_key`
- `trade_date`
- `limit`

返回：

- 当日新进入观察的标的列表
- 对应 3 / 5 / 10 日回测简表
- 若无结果则返回空数组和简单 meta

### 10.3 `GET /admin/strategy-plaza/summary`

参数：

- `strategy_key`
- `trade_date`

返回：

- 滚动窗口统计
- 回测完成度
- 规则化总结文本

### 10.4 `POST /admin/strategy-plaza/run`

用途：

- 手动重跑全部或单个策略
- 支持指定历史日期

执行方式：

- 进入现有任务队列
- 返回任务 ID

## 11. 前端页面设计

### 11.1 路由与导航

新增一级 route：

- `/strategies`

并在以下位置增加入口：

- 桌面顶栏导航
- 移动底栏导航

导航排序建议：

- `盯盘`
- `策略广场`
- `仪表盘`

### 11.2 页面结构

页面采用极简三段式：

1. 顶部工具条
   - 策略切换
   - 日期选择
   - 刷新按钮

2. 主列表
   - 当日新进入观察的标的
   - 行级展示：代码、名称、入观察日期、简短 reason、3/5/10 日表现

3. 底部摘要
   - 近 120 交易日样本数
   - 3 / 5 / 10 日胜率与均值
   - 一句总结

### 11.3 视觉约束

策略广场遵循现有交易界面的克制风格：

- 不做 hero、横幅、统计卡墙
- 不展示流程说明和使用指南
- 不把中间阶段池塞进首页
- 不让“总结”抢过主列表
- 保持低饱和配色和近不透明实底浮层

页面第一视角应该是“今天有哪些标的新进入观察”，不是“这个系统怎么工作”。

### 11.4 空态设计

首版必须支持两种空态：

1. 没有任何策略
   - 显示“暂无策略”

2. 有策略但所选日期没有结果
   - 显示“该日暂无新进入观察的标的”
   - 可附带最近有结果日期

## 12. 性能设计

### 12.1 前端性能

- 页面只读取归档结果，不现场执行策略
- 切换策略只重新拉当前策略结果
- 列表默认限制返回数量，避免一次拉全历史

### 12.2 后端性能

- 策略运行按日期批处理，不做逐条 HTTP 实时分析
- 历史补跑通过任务队列异步执行
- 回测补全只扫 `PENDING/PARTIAL` 记录
- 汇总只基于归档表聚合，不重复运行策略逻辑

### 12.3 可插拔边界

新增策略时，理论上只需：

1. 在 `builtin/` 下增加一个策略文件
2. 在注册表启用
3. 如有必要补充该策略的价格源解析

前端无须新增专用页面或专用接口。

## 13. 错误处理

首版应覆盖以下错误场景：

- 策略代码运行异常
- 某交易日输入数据不完整
- 某标的缺少锚点价格，无法生成回测记录
- 补算时未来 horizon 数据尚未可用

处理原则：

- 单个策略失败不阻断其他策略
- 单个标的回测失败不阻断同批次其他标的
- 所有失败都要落日志和错误字段
- 前端只显示简洁失败态，不暴露冗长堆栈

## 14. 测试与验证

首版至少验证以下内容：

1. 无策略时页面与接口空态正常
2. 单策略历史日期手动重跑可以正确落库
3. 同一日期重复执行不会产生重复观察记录
4. 观察结果能正确补齐 3 / 5 / 10 日回测
5. 某策略失败不会影响其他策略结果
6. 前端能在策略切换、日期切换时稳定展示
7. 每日 18:00 任务与手动任务不会打乱已归档结果

后端验证以接口调用为主，符合当前项目“通过 API + curl 验证”的开发规范。

## 15. 实施建议

首版实施顺序建议：

1. 数据表与注册表
2. 策略服务与回测补算
3. API
4. 前端一级入口与页面
5. 首个示例策略
6. 文档同步

首个示例策略可以非常简单，只要能完整走通：

- 注册
- 历史运行
- 观察结果入库
- 回测补算
- 页面展示

这样即便首版暂时只有一个简单策略，也已经具备完整的“策略广场”骨架，后续只需继续往 `builtin/` 中追加新策略。
